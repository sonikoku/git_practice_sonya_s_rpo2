# Контрольная работа: Git, GitHub и VS Code — ЗАДАНИЕ

Это **задание** к контрольной работе. Преподаватель публикует шаблон проекта на GitHub,
а вы клонируете его себе на компьютер и **продолжаете работу локально** в VS Code
по шагам ниже.

Этот файл (`TASK.md`) — инструкция, **его менять не нужно**. Свой **отчёт** вы оформляете
в отдельном файле `README.md` (в нём уже есть заготовка-шаблон) и прикладываете скриншоты
в папку `screenshots/`.

## Формат результата

GitHub-репозиторий студента + оформленный `README.md` со скриншотами в папке `screenshots/`.

## Цель работы

Закрепить полный практический цикл работы с Git и GitHub через графический интерфейс VS Code:

- вход в GitHub-аккаунт через VS Code;
- клонирование репозитория-шаблона из GitHub;
- инициализация локального Git-репозитория через `Initialize Repository`;
- добавление файлов в Stage через Source Control;
- создание осмысленных коммитов через поле `Message` и кнопку `Commit`;
- публикация проекта на GitHub через `Publish Branch`;
- отправка изменений через `Push` / `Sync Changes`;
- проверка удалённых изменений через `Fetch`;
- получение удалённых изменений через `Pull`;
- понимание разницы между `Fetch` и `Pull`;
- работа с Git Graph;
- создание и переключение веток;
- merge ветки в `main`;
- базовое разрешение конфликта в VS Code;
- настройка `.gitignore`;
- оформление `README.md` в Markdown;
- добавление скриншотов в `README.md`.

## Главное правило

Основные Git-действия выполняются **через графический интерфейс VS Code**, а не через терминал:

- Source Control;
- кнопка `+` (Stage Changes);
- поле `Message` и кнопка `Commit`;
- `Publish Branch`;
- `Push` / `Sync Changes`;
- `Fetch`;
- `Pull`;
- создание и переключение веток (панель снизу слева);
- `Merge`;
- расширение Git Graph.

Команды Git в этом задании приведены **как справка** — чтобы вы понимали, что VS Code
делает внутри. Если преподаватель требует работу именно через интерфейс, не заменяйте
её командами в терминале.

Терминал разрешается использовать только для запуска программы:

```bash
python main.py
# или
python3 main.py
```

6. После клонирования нажмите `Open`, чтобы открыть проект в VS Code.

Эквивалент команды:

## Шаг 10. Fetch, Pull и разница между ними

В этой части нужно увидеть практическую разницу: `Fetch` только **проверяет** удалённые
изменения, а `Pull` их **загружает и применяет**.

1. На GitHub откройте `README.md` → кнопка редактирования (карандаш) → внесите небольшое
   изменение (например, добавьте строку `Изменение, сделанное через GitHub`) → `Commit changes`.
2. В VS Code выполните `Fetch` (`...` в Source Control → `Fetch`).
   VS Code покажет, что в удалённой ветке появились новые коммиты,
   но **локальные файлы не изменятся**.
3. Затем выполните `Pull` — изменения с GitHub загрузятся и применятся к локальной ветке.

Эквивалент команд:

```bash
git fetch          # проверить удалённые изменения, не трогая файлы
git status         # увидеть, что ветка отстаёт на N коммитов
git pull           # загрузить и применить изменения
```

> Сделайте скриншот выполнения Fetch или Pull → `screenshots/07_fetch_or_pull.png`

## Шаг 11. Конфликт и его разрешение

Создайте и разрешите конфликт, чтобы закрепить навык.

1. **Локально**: измените одну конкретную строку в `README.md` (например, строку статуса
   `Статус проекта: ...`), сделайте коммит `docs: update status locally`.
2. **На GitHub**: измените **ту же самую строку** иначе и сделайте commit прямо на GitHub.
3. В VS Code выполните `Pull`. Так как одна строка изменена в двух местах,
   возникнет конфликт, и в файле появятся маркеры:

   ```text
   <<<<<<< HEAD
   Статус проекта: локальная версия изменена в VS Code
   =======
   Статус проекта: версия изменена через GitHub
   >>>>>>> origin/main
   ```

4. Разрешите конфликт в редакторе VS Code с помощью кнопок:
   `Accept Current Change`, `Accept Incoming Change`, `Accept Both Changes`
   или ручным редактированием.
5. Удалите все маркеры `<<<<<<<`, `=======`, `>>>>>>>`, сохраните файл.
6. Добавьте файл в Stage, сделайте коммит `fix: resolve pull conflict in readme`
   и отправьте на GitHub.

Эквивалент команд:

```bash
git pull --no-rebase     # вызовет конфликт
# ... разрешаем конфликт в редакторе ...
git add README.md
git commit -m "fix: resolve pull conflict in readme"
git push
```

> Если на занятии мало времени, конфликт можно подробно **описать** в отчёте, но за реальную
> демонстрацию даётся максимальный балл.

## Таблица Git-действий и их смысла

| Действие             | Где в VS Code                     | Смысл                                                    |
|----------------------|-----------------------------------|----------------------------------------------------------|
| Clone Repository     | Command Palette → `Git: Clone`    | Копирует репозиторий с GitHub на компьютер               |
| Initialize Repository| Source Control                    | Создаёт локальный Git-репозиторий                        |
| Stage Changes        | Source Control, кнопка `+`         | Подготавливает файлы к коммиту                           |
| Commit               | Поле `Message` + кнопка `Commit`  | Сохраняет версию проекта в истории                       |
| Publish Branch       | Source Control                    | Публикует проект/ветку на GitHub                         |
| Push                 | Source Control                    | Отправляет локальные коммиты на GitHub                   |
| Sync Changes         | Source Control                    | Синхронизирует локальные и удалённые изменения           |
| Fetch                | Source Control → `...` → `Fetch`  | Проверяет изменения на GitHub, не меняя локальные файлы  |
| Pull                 | Source Control → `...` → `Pull`   | Загружает и применяет изменения с GitHub                 |
| Create / Switch Branch| Панель снизу слева, Git Graph    | Создаёт и переключает ветки                              |
| Merge                | Git Graph / Command Palette       | Объединяет изменения одной ветки с другой                |
| Resolve Conflict     | Редактор VS Code                  | Выбор или объединение конфликтующих изменений            |
| Git Graph            | Расширение Git Graph              | Показывает историю коммитов и веток                      |

## Разница между Fetch и Pull (впишите в отчёт)

> Fetch проверяет и загружает информацию о новых коммитах из GitHub, но **не меняет**
> мои рабочие файлы. Pull загружает изменения и **сразу применяет** их к текущей ветке.
> Я проверил(а) это: изменил(а) README.md на GitHub, сделал(а) Fetch (файлы не изменились),
> а затем Pull (изменения применились).

> После Pull в файле появились маркеры конфликта. Я выбрал(а) итоговый вариант, удалил(а)
> служебные маркеры, добавил(а) файл в Stage, сделал(а) коммит решения конфликта
> и отправил(а) изменения на GitHub.

# Короткая памятка по командам (справка)

Основная работа выполняется через VS Code; команды нужны для понимания.

```bash
git clone <url>               # склонировать репозиторий
git status                    # проверить состояние проекта
git add .                     # добавить изменения в Stage
git commit -m "message"       # создать коммит
git push                      # отправить изменения на GitHub
git switch -c branch-name     # создать и перейти на ветку
git switch main               # перейти на main
git merge branch-name         # слить ветку в текущую
git fetch                     # проверить изменения, не трогая файлы
git pull --no-rebase          # забрать изменения через merge
git log --oneline --graph     # посмотреть историю коммитов
```
